## 新功能

### Windows 单机版打包

Windows 用户现在可以直接下载编译好的 zip 解压运行，不再需要 Docker 环境。发布时会在 GitHub Release 上传一份 `daidai-windows-amd64.zip`，包含下列内容：

- `daidai-server.exe`：后端主程序（Go 1.25、`CGO_ENABLED=0` 静态编译，约 21 MB）
- `ddp.exe`：同一套运维 CLI，用法与 Docker 里 `docker exec ... ddp` 完全一致
- `web/`：前端构建产物，Go 后端通过 `config.yaml` 的 `server.web_dir` 直接同端口托管，不再需要 Nginx 反代
- `config.yaml`：Windows 专用默认配置（监听 `5700`、数据目录 `./Dumb-Panel`、web_dir `./web`）
- `start.bat`：启动脚本，自带 `chcp 65001` 兜底中文控制台显示
- `README.txt`：快速上手、数据目录说明、升级方式、常见运维命令

使用步骤：

1. 从 [GitHub Release](https://github.com/linzixuanzz/daidai-panel/releases) 下载 `daidai-windows-amd64.zip`，解压到任意目录（建议路径无空格、无中文）
2. 双击 `start.bat` 启动
3. 浏览器访问 `http://localhost:5700`，首次进入创建管理员账号

首次启动会在同级目录生成 `Dumb-Panel/`（数据库、脚本、日志、备份、依赖环境），迁移/备份就是这一个目录。

**可选：脚本执行环境**。要跑 Python / Node.js 脚本的话自行安装 Python 3.10+ 和 Node.js 20 LTS，并勾选 "Add to PATH" 后重启 `start.bat`，面板会自动从 PATH 找到对应解释器。

**不包含**：面板内一键更新（Docker 专属，依赖 `docker.sock`）、Magisk 模块。Windows 升级需手工：关掉旧 `start.bat`，解压新版 zip 到新目录，把旧版本 `Dumb-Panel\` 拷进去再启动新版本。

实现要点：

- 项目很早就用 `process_windows.go` / `process_unix.go` 的 build tag 分离 `setPgid` 等 syscall 差异，本次无需新增平台适配代码，直接开 `GOOS=windows GOARCH=amd64` 交叉编译即能产出可用 exe。
- release workflow 新增 `Build Windows amd64` step（交叉编译 `daidai-server.exe` + `ddp.exe`），以及 Package step 扩展的 Windows zip 打包流程（staging 目录拷贝 exe + web + 配置 + 启动脚本 + README，`zip -rq` 打包）。
- GitHub Release 上传列表同步追加 `daidai-windows-amd64.zip`。
- Windows 专用素材放在仓库 `packaging/windows/` 目录，便于后续维护。

## 修复

### Python 依赖自动安装误报（pysmx / socks 等）——架构层修复

v2.0.7 已把预检判缺失从 `importlib.import_module()` 换成 `importlib.util.find_spec()`，本以为能覆盖所有情况。但 `pysmx` 等包在用户环境（Alpine + 托管 venv）里继续命中每一次脚本启动：

```
[检测到缺失依赖: pysmx，正在自动安装...]
Requirement already satisfied: pysmx in /app/Dumb-Panel/deps/python/venv/lib/python3.11/site-packages (0.4.0)
...
[安装成功: pysmx]
```

pip 明确说包已经在 venv 里、runpy 也能正常 `import pysmx`，唯独 **预检进程内的 `find_spec` 看不见它**。本地再加 `importlib.metadata.packages_distributions()` 和 `sys.path` 文件系统扫描两道兜底后，实测**三道防线全部漏判**——说明问题不在"某一种 Python 检测 API"，而是**预检 Python 进程启动时 sys.path 根本没包含 venv 的 site-packages**（Alpine 下 venv python 启动时对 pyvenv.cfg 的识别会出错）。

这是个架构层面的脆弱点：任何静态扫描 + Python 进程内判断的方案，都要依赖预检进程看到的 sys.path，而这一层在 Alpine venv 下不稳定。v2.0.7 一次修、本版本开发期又试过三道防线，都不是根治方案。

### 修复方式：移除预检链路，改由 Go 侧后置兜底

本版本**直接移除 bootstrap 里的 AST 扫 import + `find_spec` 预检 + 自动 pip install 全部逻辑**。Python bootstrap 现在只做三件事：

1. 从 `env.json` 注入任务环境变量
2. 把 `PYTHONPATH` 的目录前置到 `sys.path`
3. `runpy.run_path` 执行用户脚本

依赖缺失的场景由 **Go 侧 `task_executor.detectAndInstallDeps`** 兜底处理。这套机制已经存在很久：

- 脚本真实抛出 `ModuleNotFoundError: No module named 'xxx'` 或 `Cannot find module 'yyy'` 时才介入
- 调 `venv/bin/pip3 install` 精准装对应包（Python 模块名别名表仍然有效：`crypto → pycryptodome`、`yaml → pyyaml` 等 14 条）
- 装完**自动重跑脚本**，最多 5 次覆盖多依赖场景
- 同一依赖在同一次任务内不重复装（`installedDeps` 去重）

比预检更精准：100% 从**真实运行时错误**触发，不存在预检视图和实际 import 视图不一致的问题。

### 影响

- `pysmx`、`socks`、`Crypto` 等历史上被误报的包**彻底不再触发自动安装**；真正没装的包仍然能被识别并安装。
- 每次脚本启动会少一段预检耗时（对用户可感知的改善：冷启动更快、日志更干净）。
- `DD_AUTO_INSTALL_DEPS` 和 `DD_PY_AUTO_INSTALL_ALIASES` 两个环境变量不再被写入任务 env（bootstrap 不再读，留着会误导）。

### 同步修复：Python 模块名别名表补漏

预检链路移除后，后置兜底接手全部自动安装职责，**模块名 → pip 包名的别名映射**变得更关键。这次补齐了几个高频场景的映射：

| import 名 | pip 包名 | 场景 |
|----------|---------|------|
| `Cryptodome` | `pycryptodomex` | `from Cryptodome.PublicKey import RSA`——PyPI 没有叫 Cryptodome 的包，必须装带 x 后缀的兄弟包。v2.0.7 时漏了这条，脚本用 `Cryptodome.*` 的场景会直接 `ERROR: No matching distribution found for Cryptodome`。 |
| `websocket` | `websocket-client` | `import websocket`——Python 生态里 `websocket` 这个 import 名对应 PyPI 包是 `websocket-client`。 |
| `attr` | `attrs` | `import attr`——新版 `attrs` 包装后的经典写法。 |

注意：`Cryptodome` 和 `Crypto` 是**两个并行的 PyPI 包**（`pycryptodomex` vs `pycryptodome`），可以同时装在一个环境里，但 import 名不能互换。alias 表里两条都有，分别走正确的 pip 包名。

新增端到端回归测试 `TestDetectAutoInstallCandidate/python_Cryptodome_alias_maps_to_pycryptodomex`：从 `ModuleNotFoundError: No module named 'Cryptodome'` 抓取后必须解析为 `pycryptodomex`，防止这条 alias 未来再次被误删。

### 守卫测试

- 新增 `TestPythonBootstrapHasNoPreCheckAutoInstall`：扫描 `pythonEnvBootstrap` 常量，禁止再出现 `_dd_scan_imports`、`find_spec`、`packages_distributions`、`_dd_install_package`、`检测到缺失依赖` 等预检标识符。将来若有人把预检加回来，这个测试会立即失败并附带说明。
- 新增 `TestBuildManagedRuntimeEnvMapDoesNotWritePythonPreCheckEnv`：断言任务 env 里不再写入 `DD_AUTO_INSTALL_DEPS` 和 `DD_PY_AUTO_INSTALL_ALIASES`。
- 现有 `TestDetectAutoInstallCandidate`（10 个子用例，覆盖 Python/Node/Go 三类错误的抓取 + 别名映射 + 本地模块识别 + Cryptodome 回归）保持通过，保证后置兜底这条唯一链路的判缺失逻辑没被回归破坏。
