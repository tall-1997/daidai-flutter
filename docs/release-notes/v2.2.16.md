# v2.2.16

v2.2.16 是 v2.2.15 之后的稳定性修复版本，重点修复任务导入状态丢失、Python 托管虚拟环境 pip 损坏、Alpine `linux/386` 镜像构建失败，以及 Docker 面板内更新后旧镜像残留导致磁盘占用增长的问题。

## 修复

- **任务导入状态丢失** — 修复导出任务 JSON 中 `"status": 1` 的启用任务导入后全部变成禁用的问题。导入时现在会读取并校验 `status` 字段：`0` 保持禁用，`1` 保持启用；兼容历史导出值，`0.5` / `2` 会归一为启用状态。启用任务导入后，如果调度器已初始化，会自动注册对应定时规则。
- **Python 托管虚拟环境 pip 损坏** — 修复 v2.2.15 升级后旧挂载目录中的 `deps/python/venv/bin/pip3` 文件仍存在，但内部 `import pip` 失败，导致自动安装依赖时报 `ModuleNotFoundError: No module named 'pip'` 的问题。启动和安装依赖前会校验 `pip --version`，优先通过 `python -m ensurepip --upgrade` 自愈；自愈失败时会隔离损坏的 venv 并重建，随后由依赖恢复逻辑自动重装已记录的 Python 依赖。
- **Alpine Docker `linux/386` 构建失败** — Alpine 运行时不再直接使用 `node:20.19.0-alpine` 作为最终镜像底座，改为 `alpine:3.22` 并通过 `apk` 安装 Alpine 官方仓库的 `nodejs` / `npm`。这样既满足 `node >= 20.19.0` 的依赖要求，也恢复 Alpine `x86` 仓库支持，`latest` 镜像从本版本起继续支持 `linux/386` 架构构建。
- **Docker 面板内更新后旧镜像残留** — 修复面板内一键更新只拉取新镜像并重建容器、但不主动释放旧面板镜像层的问题。v2.2.16 会在新容器启动成功后，仅按更新前记录的旧面板镜像 ID 执行一次安全清理；如果旧镜像仍被其它容器使用，Docker 会拒绝删除并自动忽略，不会误清理用户其它镜像。

## 改进

- **Docker 升级磁盘占用处理** — v2.2.15 因 Alpine 运行时曾切到 `node:20.19.0-alpine`，镜像层变大属于预期变化；但宝塔面板里看不到旧容器，并不代表旧镜像层、悬空镜像或构建缓存已经释放。v2.2.16 的面板内一键更新会自动尝试清理旧面板镜像，Watchtower 部署仍建议保留 `--cleanup`。
- **Python 环境隔离增强** — 托管 Python 依赖安装时会清理可能污染虚拟环境的 `PYTHONPATH` / `PYTHONHOME`，降低升级后旧 site-packages 影响 pip、ensurepip 或依赖安装流程的概率。

## 升级说明

- 从 v2.2.15 升级到 v2.2.16 后，若检测到 Python 托管虚拟环境 pip 损坏，面板会自动尝试修复或重建 venv，用户无需手动删除 `deps/python/venv`。
- 使用 Docker 面板内一键更新时，新版本启动成功后会自动尝试清理旧面板镜像；如果宿主机上已经积累了历史悬空镜像，可在宿主机执行 `docker system df` 查看占用，并按需使用 `docker image prune -f` 清理悬空镜像。
- Alpine `latest` 镜像从本版本起使用 `alpine:3.22` + `apk` 安装 Node.js / npm，继续支持 `linux/amd64`、`linux/arm64`、`linux/386`、`linux/arm/v7`。
