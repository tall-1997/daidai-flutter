# v2.2.2

## 版本概览

`v2.2.2` 主要聚焦 Docker 自动更新链路，解决旧版 Watchtower 在新版 Docker daemon 上的兼容问题，并让面板后台能够正确识别“当前由 Watchtower 托管更新”的部署模式。

## 重点修复

### 1. Watchtower 兼容性调整

- 默认 Watchtower 方案已切换到更适合新版 Docker API 的实现：
  - `nickfedor/watchtower:latest`
- 这样可以规避部分旧版 Watchtower 在新 Docker daemon 上报错：
  - `client version 1.25 is too old. Minimum supported API version is 1.40`

### 2. 后台识别 Watchtower 托管更新

- 面板现在可以识别当前部署是否由 Watchtower 托管更新。
- 在系统设置概览中，会明确显示：
  - `更新方式：Watchtower 托管更新`
- 同时会展示 Watchtower 的调度信息和相关提示，不再把 Watchtower 模式误判成普通“容器内自更新失败”。

### 3. 预留手动触发 Watchtower 检查能力

- 生产版配置已预留：
  - `WATCHTOWER_HTTP_API_TOKEN`
  - `WATCHTOWER_HTTP_API_URL`
  - `WATCHTOWER_HTTP_API_PERIODIC_POLLS`
- 面板后端已补上 Watchtower 手动触发能力的接口骨架，当前部署一旦正确启用 HTTP API，即可通过后台触发一次 Watchtower 检查更新。

### 4. Docker Compose / docker run / 文档统一

- 基础版与生产版 Compose 都已同步切换到新的 Watchtower 镜像。
- `docker run` 示例也已更新为与 Compose 一致的 Watchtower 托管方案。
- README 文档同步补充：
  - Watchtower 托管模式说明
  - Push Plus 通知接入
  - HTTP API 手动触发说明

## 测试与验证

- 本次为 Docker 更新链路与配置修复版本，主要完成：
  - 后端 Watchtower 托管识别逻辑接入
  - 前端概览页 Watchtower 模式文案与按钮切换
  - Compose / 环境变量 / README 配置一致性检查

## 升级说明

- 推送 Git tag `v2.2.2` 后将自动触发 GitHub Release 工作流。
- 如果你已经部署了旧版 Watchtower，建议升级到新的 Compose 配置后重新拉起：
  - `docker compose pull watchtower`
  - `docker compose up -d watchtower`
- 如果你希望后台支持“手动触发 Watchtower 检查更新”，请额外配置：
  - `WATCHTOWER_HTTP_API_TOKEN`
  - `WATCHTOWER_HTTP_API_URL`
